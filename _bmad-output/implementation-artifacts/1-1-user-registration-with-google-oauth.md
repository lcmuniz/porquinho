# Story 1.1: User Registration with Google OAuth

Status: done

## Story

As a new user,
I want to register for an account using my Google account,
So that I can quickly sign up without creating a new password.

## Acceptance Criteria

**Given** I am on the registration page
**When** I click "Sign up with Google" button
**Then** I am redirected to Google OAuth consent screen
**And** After granting permissions, I am redirected back to the application
**And** A new user account is created with my Google email and profile info
**And** A `users` table is created via Flyway migration (V1__create_users_table.sql) with fields: user_id (UUID, PK), email, auth_provider, google_id, created_at, updated_at, deleted_at, deleted_by
**And** User record is inserted with auth_provider='google', google_id populated
**And** JWT token is generated by Supabase Auth and stored in browser localStorage (NFR14 - adapted for Supabase architecture)
**And** I am logged in and redirected to onboarding page
**And** Audit log entry is created for "user_registration" event (NFR20)
**And** LGPD consent is recorded with timestamp (NFR25)

## Dev Notes

### Critical Architecture Context

**Authentication Flow (Supabase Auth + Spring Boot JWT Validation):**

1. Frontend → Supabase Auth (signInWithOAuth)
2. Supabase → Google OAuth → User consent
3. Supabase returns JWT token (access_token, refresh_token)
4. Frontend stores token (Supabase handles this in localStorage)
5. Frontend sends token in Authorization header: `Bearer {access_token}`
6. Spring Boot validates JWT via JWKS endpoint
7. Spring Security extracts user_id from "sub" claim
8. Controllers use `@AuthenticationPrincipal String userId`

**Database Schema (PostgreSQL via Supabase):**
- Primary key: `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`
- Foreign keys: `{table_singular}_id` format (e.g., `user_id`)
- Audit columns: `created_at`, `updated_at`, `deleted_at`, `deleted_by`
- Soft deletes: Use `deleted_at IS NULL` in queries
- Naming: `snake_case` for tables/columns

**API Design (RESTful):**
- Base path: `/api/v1/*`
- Endpoints: Plural nouns (`/users`, `/transactions`)
- Headers: `Authorization: Bearer {jwt}`, `Content-Type: application/json`
- Error responses: RFC 7807 Problem Details format

**Frontend Stack:**
- Vue 3.5 + Vite 7 + TypeScript
- shadcn-vue (reka-ui ^2.9.2 headless components)
- Tailwind CSS v4 CSS-first (NO tailwind.config.js)
- Pinia for state management
- Axios for HTTP requests

### Security Requirements

**NFR13 - Password Hashing:** Not applicable for Google OAuth (no password stored)

**NFR14 - JWT Storage (Architectural Decision):** Supabase Auth stores JWT tokens in browser localStorage (not httpOnly cookies). This is the standard Supabase architecture pattern. While localStorage is vulnerable to XSS attacks, this is mitigated by:
- Content Security Policy (CSP) headers
- React/Vue framework's built-in XSS protection
- Regular security audits
- Supabase's automatic token rotation
**Alternative considered:** Backend cookie proxy - adds complexity without significant security benefit given Supabase's security model. Decision: Accept Supabase's localStorage pattern as industry-standard for OAuth flows.

**NFR16 - Session Expiration:** Supabase JWT expires after 7 days. Frontend refresh token logic handles renewal.

**NFR17 - Rate Limiting:** Not implemented in Story 1.1 (add in Story 1.3 - Login)

**NFR20 - Audit Logging:** MUST create audit log entry for "user_registration" with timestamp and IP address

**NFR25 - LGPD Consent:** MUST record `lgpd_consent_at` timestamp on user registration
